On-Screen Keyboard Research

Responses from a Power User Perspective

Owen Kent

1. Your Relationship to the On-Screen Keyboard

I use the Windows on-screen keyboard as a primary input method, not a fallback. It is something I rely on daily for communication, development, creative work, and basic system navigation. For me, it is not optional.

I typically use it alongside alternative inputs like switches, joysticks, dwell-based pointing, virtual mouse systems, and custom accessibility controllers. These systems work together as a layered stack, but the on-screen keyboard is still the bottleneck.

2. Design Quality and "Modern Windows" Fit

The on-screen keyboard does not feel like a modern, first-class part of Windows. It feels like a legacy utility that has been visually refreshed just enough to not stand out, but not fundamentally redesigned.

There are frequent inconsistencies across apps, input fields, and system contexts. Focus handling is unreliable, layout scaling feels brittle, and interaction patterns often clash with modern Windows UI paradigms.

It feels bolted on rather than deeply integrated.

3. Overall Impressions

Emotionally, using the on-screen keyboard is frustrating and fatiguing. It works just well enough to get by, but not well enough to feel empowering. Over time, that erodes confidence and independence.

It does not feel like people who rely on it were genuinely centered in its design. It feels designed to satisfy a checkbox for accessibility compliance rather than to support high-throughput, long-duration, real-world use.

There is a big difference between "technically accessible" and "designed for."

4. What Works Well Today

The fact that the on-screen keyboard exists system-wide and works across most applications is valuable. Basic predictability and persistence matter, and I rely on that baseline stability.

I also rely on consistent key placement and the ability to invoke it reliably through accessibility settings or shortcuts. Those fundamentals should not be lost.

Stability matters, especially for users who build muscle memory over years.

5. Missing Features (Part 1)

Word prediction is the biggest missing capability. Modern text entry should not require spelling every word from scratch, especially for users who type slowly or with high physical effort.

Prediction, next-word suggestion, phrase learning, and adaptive language models should be core features, not afterthoughts.

Federated Learning: My data should stay on my device, while still contributing to better models across users. That approach respects privacy while allowing systems to improve meaningfully over time.

5. Missing Features (Part 2)

UI customization is severely lacking. I want control over:

  • Layout and spacing
  • Key size and grouping
  • Colors and contrast
  • Interaction modes

Different users need radically different configurations, and a one-size-fits-all keyboard is fundamentally incompatible with accessibility.

Many tasks feel harder than they should, especially long-form writing, coding, command-line work, and creative workflows.

6. Friction, Annoyances, Customization & Control

The biggest sources of friction are inefficiency and rigidity. Every extra keystroke, every mis-registered focus change, and every unnecessary mode switch adds up. Over time, that causes real physical fatigue.

I have very little control over how the keyboard adapts to me. Instead, I am forced to adapt to it.

There is minimal behavioral learning, limited personalization, and almost no extensibility. For such a critical assistive tool, that lack of control is unacceptable.

7. Closing Reflection

If I could explain one thing to a Windows designer, it would be this:

The on-screen keyboard is not a niche feature. For many of us, it is the computer.

Treat it with the same seriousness as the desktop shell, the file system, or the browser.

What "Excellent" Would Look Like

A modular, highly customizable, open-architecture on-screen keyboard with:

  • Built-in word prediction
  • Adaptive learning
  • Strong privacy guarantees
  • Plugin support
  • Alternative layouts
  • Community-driven experimentation

It would invite an open-source ecosystem rather than walling it off.

Design for Real Use

Most importantly, it would be designed with the assumption that some users will spend eight to twelve hours a day using it, every day, for years.

That perspective changes everything.

A Note on Context

This critique is not just frustration. It is architectural.

This is a reaction to a missed opportunity to turn one of the most important accessibility tools in computing into a platform for innovation, community, and real independence.

As an engineer, founder, filmmaker, and disability rights advocate who builds systems when the existing ones fall short, I think in terms of ecosystems, not features.
