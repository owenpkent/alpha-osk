## 1. Your Relationship to the On-Screen Keyboard

I use the Windows on-screen keyboard as a primary input method, not a fallback. It is something I rely on daily for communication, development, creative work, and basic system navigation. For me, it is not optional. If it is slow, inaccurate, or poorly designed, my entire computing experience degrades.

I typically use it alongside alternative inputs like switches, joysticks, dwell-based pointing, virtual mouse systems, and custom accessibility controllers. These systems work together as a layered stack, but the on-screen keyboard is still the bottleneck. No matter how advanced my input hardware is, everything eventually funnels through text entry. That makes the keyboard one of the most critical pieces of assistive infrastructure on the system.

---

## 2. Design Quality and “Modern Windows” Fit

The on-screen keyboard does not feel like a modern, first-class part of Windows. It feels like a legacy utility that has been visually refreshed just enough to not stand out, but not fundamentally redesigned. In many ways, it still behaves like it did in the Windows XP era.

There are frequent inconsistencies across apps, input fields, and system contexts. Focus handling is unreliable, layout scaling feels brittle, and interaction patterns often clash with modern Windows UI paradigms. In short, it feels bolted on rather than deeply integrated. That is a problem when this tool is foundational to how some of us use the operating system at all.

---

## 3. Overall Impressions

Emotionally, using the on-screen keyboard is frustrating and fatiguing. It works just well enough to get by, but not well enough to feel empowering. Over time, that erodes confidence and independence.

It does not feel like people who rely on it were genuinely centered in its design. It feels designed to satisfy a checkbox for accessibility compliance rather than to support high-throughput, long-duration, real-world use. There is a big difference between “technically accessible” and “designed for.”

---

## 4. What Works Well Today

The fact that the on-screen keyboard exists system-wide and works across most applications is valuable. Basic predictability and persistence matter, and I rely on that baseline stability.

I also rely on consistent key placement and the ability to invoke it reliably through accessibility settings or shortcuts. Those fundamentals should not be lost. Stability matters, especially for users who build muscle memory over years.

---

## 5. Missing Features

Word prediction is the biggest missing capability. Modern text entry should not require spelling every word from scratch, especially for users who type slowly or with high physical effort. Prediction, next-word suggestion, phrase learning, and adaptive language models should be core features, not afterthoughts.

I am particularly interested in federated learning here. My data should stay on my device, while still contributing to better models across users. That approach respects privacy while allowing systems to improve meaningfully over time.

UI customization is also severely lacking. I want control over layout, spacing, key size, grouping, colors, contrast, and interaction modes. Different users need radically different configurations, and a one-size-fits-all keyboard is fundamentally incompatible with accessibility.

Many tasks feel harder than they should, especially long-form writing, coding, command-line work, and creative workflows. These are things I do professionally, and the keyboard actively slows me down.

---

## 6. Friction, Annoyances, Customization & Control

The biggest sources of friction are inefficiency and rigidity. Every extra keystroke, every mis-registered focus change, and every unnecessary mode switch adds up. Over time, that causes real physical fatigue.

I have very little control over how the keyboard adapts to me. Instead, I am forced to adapt to it. There is minimal behavioral learning, limited personalization, and almost no extensibility. For such a critical assistive tool, that lack of control is unacceptable.

From a technical perspective, it is one of the least customizable pieces of assistive technology in the Windows ecosystem, which is deeply ironic given how central it is.

---

## 7. Closing Reflection

If I could explain one thing to a Windows designer, it would be this: the on-screen keyboard is not a niche feature. For many of us, it *is* the computer. Treat it with the same seriousness as the desktop shell, the file system, or the browser.

“Excellent” would look like a modular, highly customizable, open-architecture on-screen keyboard with built-in word prediction, adaptive learning, and strong privacy guarantees. It would support plugins, alternative layouts, and community-driven experimentation. It would invite an open-source ecosystem rather than walling it off.

Most importantly, it would be designed with the assumption that some users will spend eight to twelve hours a day using it, every day, for years. That perspective changes everything.
